# Phase 1 Research: Ingestion, Versioning & CI/CD Foundation

**Objective:** Что нужно знать, чтобы хорошо спланировать Phase 1 (ING-01/02/03, VER-01/02/03, CICD-01/02).
**Source context:** 01-CONTEXT.md + REQUIREMENTS.md + STATE.md.

## 1. Зафиксированные решения (не обсуждать повторно)
- Детерминизм ingestion не строгий: качество важнее бит?в?бит идентичности при обновлениях парсера.
- Метаданные v1: минимум (источник, дата, автор/владелец, раздел/страницы).
- OCR для сканов: best?effort с отметкой качества.
- Частичный фейл ingestion: fail?fast + подробный отчёт ошибок.
- Инкрементальные обновления: батчи, расписание; изменение по контент?хешу; удалённые документы удаляются; целевой интервал — ежедневно.
- Версии: auto name = timestamp + hash; активную версию переключает только админ; rollback мгновенно на предыдущую; хранить последние 3 версии.
- CI/CD: блокирующие проверки pytest + ruff; main защищён, merge через PR; CI на каждый PR = тесты + линт; автодеплой не нужен.
- Кодовой базы пока нет.

## 2. Что критично прояснить ДО планирования
Ниже — решения, без которых план будет рыхлым или рискованным.

### 2.1. Среда и стек
- Язык/рантайм и структура репозитория (Python/poetry/uv/requirements? mono?repo или single?service?).
- Целевая ОС/среда для запуска ingestion (локально/сервер/контейнер). Это влияет на OCR и библиотеки парсинга.
- Минимальный набор зависимостей разрешён? (избегать системных бинарей или можно ставить Tesseract/Poppler).

### 2.2. Источники и формат входных данных

### 2.3. Стратегия chunking и стабильность
- Правило chunking (по страницам, по заголовкам, по токенам, смешанное).
- Политика «стабильности» чанков при незначительных изменениях документа (например, small edit не должен менять все chunk_id).
- Формат хранения метаданных на чанк (что именно и где хранится).

### 2.4. Версионирование и идентификация
- Что именно является «версией» на диске/в БД (dataset_version, embedding_version, index_version — отдельные сущности?).
- Где хранить маппинг «активная версия» (конфиг/БД/файл?указатель).
- Как «складываются» версии (каталоги, префиксы, таблицы) и как чистить старые (TTL/retain?3).

### 2.5. Инкрементальная загрузка
- Источник правды для content?hash (файл целиком vs извлечённый текст). Разные варианты меняют воспроизводимость.
- Что считается «изменённым»: смена метаданных? изменение OCR? изменение чанков?
- Поведение при конфликте: если документ изменился, пересоздавать только его chunks/embeddings или ещё и индексы.

### 2.6. Минимальная схема хранилищ
- Нужна ли уже в Phase 1 отдельная БД для метаданных (SQLite/Postgres) или можно файловую структуру + manifest.
- Где хранить артефакты: raw_text, chunks, embeddings, manifest.

### 2.7. CI/CD
- Где хостится репозиторий (GitHub/GitLab/Bitbucket) > выбор CI.
- Политика ветвления (main/develop), требуемые статусы для merge.
- Минимальный набор тестов в Phase 1 (unit + e2e для ingestion).

## 3. Референс?варианты (на что ориентироваться при планировании)
Не решения, а «рамки», чтобы план был реализуемым.

### 3.1. Архитектура ingestion v1 (минимальная)
- Вход: PDF(ы) > извлечение текста > chunking > метаданные + manifest.
- Выход: детерминированные chunk_id, стабильные метаданные, reproducible manifest.
- Инструмент запуска: простой CLI (например `ingest --input <path> --dataset <name>`), чтобы можно было прогонять в CI.

### 3.2. Версионирование
- `dataset_version`: фиксирует список документов и их content?hash.
- `embedding_version`: фиксирует модель + параметры.
- `index_version`: связывает dataset_version + embedding_version.
- Активная версия: один файл?указатель или запись в таблице `active_index`.

### 3.3. Инкрементальная логика
- `doc_hash`: либо hash PDF bytes, либо hash normalized text.
- Если doc_hash совпадает > пропуск.
- Если изменён > пересоздать chunks/embeddings только для него.
- Если удалён > удалить его чанки/эмбеддинги из текущей версии.

### 3.4. CI/CD
- Pipeline: `lint` (ruff) + `test` (pytest) на каждый PR/push.
- Отдельная job для E2E ingestion с маленьким фикстурным PDF.

## 4. Риски и как их учесть в плане
- Недетерминизм парсеров PDF и OCR: нужно фиксировать версии библиотек, настройки и seed, иначе ING?02 нарушается.
- Разная структура PDF > chunking может «плавать»: нужен формат manifest и стабильные правила.
- OCR системные зависимости > CI может не иметь бинарей (нужен контейнер/докер/skip?path).
- Отсутствие БД может усложнить versioning/rollback > хотя бы manifest?store должен быть транзакционным (atomic write).

## 5. Минимальные артефакты, которые план должен выпускать
- Структура storage (папки/таблицы) для raw/chunks/embeddings/manifest.
- Формат manifest (JSON/CSV/Parquet) с schema и version_id.
- CLI/скрипт ingestion (воспроизводимая команда).
- Тестовые фикстуры PDF + snapshot?тесты на manifest/chunks.
- CI config, запускающий lint/test.

## 6. Что нужно решить прямо перед началом планирования (короткий чек?лист)
- Где будет храниться metadata/manifest (файлы vs БД).
- Чем парсим PDF и как обрабатываем сканы (системные зависимости допускаются?).
- Формат и стабильность chunk_id.
- Схема версий и хранение активной версии.
- Где и как запускать CI.

## Validation Architecture
Ниже — каркас валидации, который можно перенести в VALIDATION.md для Nyquist.

- **Framework:** pytest (ожидается) + ruff для линта.
- **Quick run:** `pytest -q` (без тяжелых OCR/ингест-фикстур).
- **Full suite:** `pytest` + `ruff check .`
- **Estimated runtime:** 30–90s на маленьких PDF-фикстурах.
- **Wave 0:** добавить базовые фикстуры PDF + тесты snapshot-стабильности чанков/manifest.
